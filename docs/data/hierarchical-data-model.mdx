Also known as `HDM`

### Pros

`-` Data can be retrieved easily due to the explicit links between the table structure <br />
`-` Referential integrity is always maintained. i.e. any changes made in the parent table are automatically updated in a child table <br />
`-` Promotes data sharing <br />
`-` It is conceptually simple due to the parent-child relationship <br />
`-` Database security is enforced <br />
`-` Quick access at root nodes <br />
`-` Efficient with 1:N relationships <br />
`-` A clear chain of command or authority <br />
`-` Increases specialisation <br />
`-` Clean results <br />
`-` High performance <br />

### Cons

`-` If the parent table and child table are unrelated, then adding a new entry in the child table is difficult because affitional entries muct be added in the parent table <br />
`-` Complex relationships are not supported <br />
`-` Redundancy which results in inaccurate information <br />
`-` M:N relationships are not supported <br />
`-` No data manipulation or data definition language  <br />
`-` Lack or standards  <br />
`-` Poor flexibility <br />
`-` Difficult to implement into a database  <br />
`-` Organisational disunity
`-` Rigid structure

## What does the `N` mean?
`N` mean **many** <br />
Example: <br />
M:N = Many to Many
